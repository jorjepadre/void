# Django Patterns

Django for maintainable web applications: models, views, ORM, and testing.

## Project Structure

- Split monolithic apps into focused apps around bounded contexts.
- `settings/` package with `base.py`, `dev.py`, `prod.py`, `test.py`.
- Keep secrets in env vars, loaded via `django-environ` or `os.environ`.

## Models

- Define domain invariants in `clean()`, enforce with `full_clean()` on save.
- Custom `Manager`/`QuerySet` for reusable query logic:

```python
class ActiveUserQuerySet(models.QuerySet):
    def active(self):
        return self.filter(is_active=True, deleted_at__isnull=True)

class User(models.Model):
    objects = ActiveUserQuerySet.as_manager()
```

- Use `choices = models.TextChoices` enums for enumerated fields.
- Indexes declared in `Meta.indexes`; review migrations for unintended table scans.

## ORM

- `select_related` for ForeignKey/OneToOne; `prefetch_related` for ManyToMany/reverse FK.
- Use `only`/`defer` to limit columns on hot paths.
- `bulk_create`/`bulk_update` for batch writes.
- Avoid N+1 — inspect queries with `django-debug-toolbar` or `django-silk`.

## Views

- Class-based views for standard CRUD; function-based for unique flows.
- Keep view logic thin — delegate to services/use-cases.
- DRF ViewSets + serializers for APIs; plain views + forms for HTML.

## Forms and Serializers

- Validate at the edge; never trust request data.
- Serializer `validate_*` methods for field-level rules; `validate` for cross-field.

## Admin

- Register models with `list_display`, `list_filter`, `search_fields`.
- `raw_id_fields` for FKs to large tables to avoid dropdown blowups.

## Async

- Django 4.1+ supports async views, ORM (`.aall`, `.afirst`).
- Use Celery/RQ for background tasks; don't block request/response.

## Testing

- `pytest-django` with fixtures; factory_boy for model factories.
- Use `TestCase` (DB transactions rolled back per test) or `TransactionTestCase` when needed.
- Test the happy path, one error per branch, and edge cases.
- `Client` for view tests; avoid testing through layers of middleware for unit-level tests.

## Security

- `SECURE_*` settings, `CSRF_COOKIE_SECURE`, `SESSION_COOKIE_SECURE` in production.
- Use `django-csp` for Content Security Policy.
- Rate-limit auth endpoints.
- Never log PII or secrets.

## Migrations

- Review autogenerated migrations; rename/squash before deploy.
- Separate schema migrations from data migrations.
- For zero-downtime deploys: add nullable columns, backfill, then make non-null.

## Common Pitfalls

- `related_name` collisions across apps — always set explicitly.
- `auto_now_add` vs `auto_now` confusion.
- Raw SQL that bypasses ORM protections.
- Circular imports between models — use string references in FKs.
